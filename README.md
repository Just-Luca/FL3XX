# BRIGHT Aviation BI — Portfolio Project

Portfolio project for the FL3XX BRIGHT role — full BI engineering workflow from data modelling and SQL analytics to an interactive dashboard, automated data quality monitoring, and QuickSight deployment prep. Built with synthetic aviation data.

---

## Honest scope statement

This project uses **fully synthetic data**. There is no connection to FL3XX's systems, no real operator data, and no access to Amazon QuickSight in production. Everything here — the flights, the aircraft, the crew, the revenue figures — was generated programmatically to mimic what a real charter operator's data warehouse might look like.

The goal is not to simulate FL3XX's actual platform. The goal is to demonstrate, concretely and end-to-end, the exact skills the role requires: schema design, SQL analytics, BI dashboard construction, data quality monitoring, and AWS QuickSight preparation. Every technical decision was made with that job description in mind.

What this project **is**:
- A working demonstration of the full BI engineering workflow, from raw data to published dashboard
- A learning artefact built by someone coming from a data science / forecasting background, deliberately acquiring new BI engineering skills
- An honest attempt to speak the language of aviation operations (fleet utilization, FTL compliance, yield per block hour, operator isolation)

What this project **is not**:
- A production system
- Real aviation data
- A replacement for hands-on QuickSight experience (Phase 5 provides the scaffolding; actual console work requires an AWS account)

## Phase 1 — Data generation & schema design

**File:** `generate_data.py`

Builds a star schema representing two years of charter flight operations (2023–2024) for a fictional mid-size aviation group running four operators: a charter company, an aircraft management firm, a fractional provider, and a medevac operator.

**Schema design:**

```
FACT_FLIGHTS  (3,030 rows — one row per flight)
    │
    ├── DIM_AIRCRAFT   (8 rows  — tail number, type, range, capacity)
    ├── DIM_CREW       (20 rows — name, role, duty hours YTD, FTL status)
    ├── DIM_CLIENT     (12 rows — name, type: corporate/broker/leisure/cargo)
    ├── DIM_ROUTE      (12 rows — ICAO pairs, distance, region)
    └── DIM_OPERATOR   (4 rows  — operator name and type)
```

Key design decisions:
- **Grain:** one row = one flight. Revenue, block hours, delay, and status are flight-level metrics.
- **operator_id on every flight:** this is the foundation of multi-tenant Row-Level Security — every downstream filter keys off this column.
- **Intentional data quality issues injected:** ~30 duplicate rows, ~15 flights with NULL revenue, 3 flights with impossible block hours (99.0h), 2 crew members with FTL violations. These are detected and documented in Phase 4.

**Key concept — grain:** defining "one row = one flight" before writing any code is the most important modelling decision. If you get the grain wrong, every aggregate metric downstream is wrong.

## Phase 2 — SQL analytics layer

**File:** `analytics.py`  
**Engine:** DuckDB (Redshift-compatible SQL dialect, runs locally with no server)

Builds a `v_flights` view that deduplicates, excludes bad data, joins all dimensions, and adds derived fields. All analytical queries read from this view — never from raw tables.

**Queries implemented:**

| # | Query | SQL techniques |
|---|-------|---------------|
| Q1 | Fleet utilization % by tail and year | GROUP BY, derived utilization calc |
| Q2 | Monthly revenue + 3-month rolling average | `AVG() OVER (ROWS BETWEEN 2 PRECEDING...)` |
| Q3 | Revenue per operator with rank and share | `RANK() OVER`, `CROSS JOIN` for totals |
| Q4 | On-time performance by route and region | Conditional aggregation |
| Q5 | Crew duty compliance and FTL flag | `CASE WHEN` duty status classification |
| Q6 | Client YoY revenue growth | `LAG() OVER (PARTITION BY client ORDER BY year)` |
| Q7 | Operator-isolated view (RLS simulation) | `WHERE operator_id = ?` |
| Q8 | Data quality audit trail | Counts excluded rows and explains why |

**Key concept — the view layer:** `v_flights` sits between raw tables and analytical queries. Cleaning logic lives here once; all downstream tools consume the clean surface. In a real Redshift setup, QuickSight points at this view.

## Phase 3 — Interactive BI dashboard

**File:** `dashboard.py`  
**Stack:** Streamlit + Plotly + DuckDB

A six-section dashboard that mirrors the kind of embedded analytics FL3XX's BRIGHT platform exposes to operators. The sidebar operator filter simulates QuickSight's Row-Level Security — every SQL query is dynamically rewritten to include `WHERE operator_id = ?` based on the selected operator.

**Dashboard sections:**

| Section | Content |
|---------|---------|
| 01 KPIs | Total flights, block hours, revenue, on-time %, avg yield, avg delay |
| 02 Fleet utilization | Bar chart vs 75% target, block hours vs revenue scatter |
| 03 Revenue trend | Monthly bar + 3M rolling average line, client revenue donut |
| 04 Delay & OTP | On-time % by route, avg delay vs OTP scatter |
| 05 Crew & FTL | Duty hours bar with breach alerts, flights vs block hours scatter |
| 06 Client YoY | Year-on-year revenue growth by client |

**To run locally:**
```bash
pip install streamlit plotly duckdb pandas
streamlit run dashboard.py
# opens at http://localhost:8501
```

**Key concept — `@st.cache_resource`:** Streamlit reruns the entire script on every interaction. Caching the DuckDB connection and query results prevents reconnecting and re-querying on every filter change.

## Phase 4 — Data quality & ETL monitoring

**File:** `data_quality.py`  
**Output:** `data_quality_report.pdf`

Runs 10 automated checks against the raw data layer, classifies each by severity, and generates a PDF report — the "one-pager to management" the job description explicitly asks for.

**Checks implemented:**

| Severity | Check |
|----------|-------|
| CRITICAL | NULL revenue on completed flights |
| CRITICAL | Impossible block hours (IQR outlier detection) |
| CRITICAL | FTL compliance breaches in crew dimension |
| WARNING  | Duplicate flight records |
| WARNING  | Revenue statistical outliers (3-sigma) |
| WARNING  | Cancelled flights with non-zero revenue |
| WARNING  | Crew double-booked on same date |
| OK       | Orphaned foreign keys |
| OK       | Flights outside expected date range |
| INFO     | Data freshness (days since latest flight) |

Each check returns: what is wrong, how many rows are affected, and a concrete recommendation. The PDF report is designed to be sent automatically after each ETL run.

**Key concept — severity classification:** not all data issues are equal. A NULL revenue on a completed flight is a billing risk (CRITICAL). A duplicate row inflates metrics but doesn't lose money (WARNING). Classifying by business impact, not just pass/fail, is what makes a monitoring report actionable.

## Phase 5 — QuickSight preparation (AWS)

**File:** `quicksight_prep.py`  
**Output:** `quicksight/` folder

Prepares the clean dataset for deployment to Amazon QuickSight — the actual BI tool used in the FL3XX BRIGHT platform. Generates a clean fact CSV, an S3 manifest file, an RLS rules file, and a step-by-step console guide.

**When AWS credentials are configured** (`aws configure`), the script:
1. Creates an S3 bucket
2. Uploads all clean CSVs
3. Generates the QuickSight datasource manifest
4. Prints exact ARNs and console URLs

**Without AWS credentials**, it runs in dry-run mode and generates all local files for review.

**Row-Level Security file (`rls_operator_rules.csv`):**
```
UserName                          operator_id
operator.alphajet@company.com     1
operator.eurowing@company.com     2
operator.skyfrac@company.com      3
operator.medair@company.com       4
```
In QuickSight, this CSV is uploaded as a second dataset and attached to the main dataset as a GRANT-type RLS rule. Each user then sees only their operator's flights — automatically, at query time, without any dashboard-level filtering logic.

**Key concept — RLS is a dataset, not a setting:** QuickSight enforces row-level access by joining the RLS CSV to every query at runtime. Adding a new operator user means updating the CSV — it's a data operation, not an admin one.
